# SP3.5 — LLM Policy Authoring — Design Spec

**Status:** Approved (2026-06-26).
**Depends on:** SP1 (audit chain), SP2 (policy IR + enforcer), SP3 (model provider).
**Delivers:** natural-language operator intent → a strictly-validated, tool-grounded `PolicySchema` proposal, with a two-step human-gated propose→activate flow and tamper-evident activation provenance.

---

## 1. Decisions locked (from brainstorming)

| Question | Decision |
|---|---|
| How does the model produce structured policy reliably? | **OpenAI JSON mode** (`response_format={"type": "json_object"}`) via a `generate(prompt) -> str` method on the existing `ModelProvider` seam, so authoring is swappable too. JSON mode only guarantees the body parses; correctness comes from two post-parse gates (`PolicySchema` + tool grounding). |
| Human-in-the-loop activation | **Two-step propose → activate.** The engine returns a validated `PolicyProposal` + diff (testable core); a thin CLI stages it (`policies/proposed/`) and a separate explicit `activate` promotes it into `policies/`. Nothing auto-loads. |
| Input contract & field scope | **Grounded.** A tool inventory is required and listed in the prompt; a post-parse grounding gate rejects any referenced tool not in it. Authors `denylist`, `external_comms_tools`, `tool_rules`, `taint_sources`, `content_scan_sources`. **Not** `injection_signatures` (curated separately). |

**Why the model's output is still safe here:** the model only *proposes*; a human *disposes*. Two validation gates reject malformed or hallucinated output, and nothing activates without an explicit human step. The architecture is the asset; the model is a swappable stand-in behind the `ModelProvider` seam.


**Policy-IR extraction (refactor):** `warden/authoring` is the second consumer of `PolicySchema`/`load_policy` (the enforcer is the first), so the IR moves to its own `guard/policy.py`. `enforcer.py` imports it; every SP2 test stays green. Same justified-by-a-second-consumer extraction as the SP3 Aho-Corasick move. No behavior change.

## 3. Provider extension (`models/provider.py`, `openai_provider.py`)

`ModelProvider` gains:
```python
@abstractmethod
async def generate(self, prompt: str) -> str: ...
```
`OpenAIGuardProvider.generate` calls `chat.completions.create(model=settings.openai_guard_model, messages=[{"role": "user", "content": prompt}], response_format={"type": "json_object"}, temperature=0.0, max_tokens=settings.openai_author_max_tokens)` and returns the message text. JSON mode only guarantees the body parses; the authoring pipeline's two gates are the real correctness guarantee. Any client error fails closed via `InferenceError`. The native call is exercised through a fake chat client (no network), mirroring `assess`. The test `StubProvider` gains a scripted `generate`.

> Library-syntax note: the JSON-mode `response_format={"type": "json_object"}` shape is per the documented OpenAI Python SDK (`openai>=1.40`). The exact field access is confirmed against the installed SDK at implementation. No method signature is invented.

## 4. The authoring pipeline (`warden/authoring.py`) — defense in depth

`async author_policy(provider, intent, *, tools, current_policy) -> PolicyProposal`:

1. **Prompt** from the inventory (`_build_author_prompt`): the tool list, the operator intent, and the required JSON shape (the five fields, with values drawn from the verdict set `ALLOW`/`BLOCK`/`HUMAN_APPROVAL_REQUIRED` and the tier set `LOW`/`MEDIUM`/`HIGH`).
2. **Generate** a JSON object via `provider.generate` (OpenAI JSON mode).
3. **Parse** — `json.loads`; malformed ⇒ `AuthoringError` (fail-closed).
4. **Structural gate** — `PolicySchema.model_validate` (types + `extra="forbid"`).
5. **Grounding gate** — every tool referenced (in any of the five fields) must be in the inventory, else `AuthoringError`. This is the gate that catches a hallucinated tool name.
6. **Diff** vs `current_policy` → `PolicyDiff`.
7. Return frozen `PolicyProposal(policy, diff, intent)`.

The proposal is a **delta fragment** that merges via `load_policy` (the existing merge model), not a full replacement.

### Data contracts
- `PolicyProposal` (frozen): `policy: PolicySchema`, `diff: PolicyDiff`, `intent: str`.
- `PolicyDiff` (frozen): per-field additions and changes vs the current merged policy (lists: new entries; dicts: new keys and `old→new` value changes), with a human-readable `render()`.

## 5. Staging & activation (`scripts/policy_author.py`)

- `propose "<intent>"` — runs the pipeline against the active policy + a configured tool inventory, writes the validated proposal to `policies/proposed/<slug>.yaml` with the **intent and provenance as a header comment**, and prints the diff. Nothing is active.
- `activate <file>` — re-loads and re-validates the staged file (it must still parse as policy), promotes it into `policies/`, and appends a **tamper-evident activation record to the SP1 audit chain** (`AuditRecord`, `verdict=ALLOW` as a neutral marker, `details = {event: "policy.activated", intent, policy_sha256}`). "Who changed the rules, from what intent, when" becomes provable — the Verifiable-Enforcement thesis applied to governance.

---

## 6. Settings / exceptions

- Settings: `openai_author_max_tokens: int = 1024` (a generated policy proposal is larger than a verdict), `proposed_policy_dir: Path = policies/proposed`.
- New exception `AuthoringError(SecureSGError)` for any generation/parse/validation/grounding failure. The pipeline never yields a partial or unvalidated policy.

---

## 7. Error handling / fail-closed

- Malformed JSON, schema-invalid output, or any reference to a tool outside the inventory ⇒ `AuthoringError`; no proposal is produced.
- A staged file that no longer parses ⇒ `activate` refuses and raises.
- Activation auditing shares the SP1 chain's atomic, idempotent append.

## 8. Testing

`StubProvider.generate` returns scripted JSON — no network. Coverage:
- `_build_author_prompt` (pure): the built prompt lists the tool inventory and names the verdict and tier value sets.
- `authoring.py`: happy path → valid `PolicyProposal`; malformed JSON → `AuthoringError`; schema-invalid → `AuthoringError`; **hallucinated/ungrounded tool → `AuthoringError`**; diff correctness (additions + value changes).
- `openai_provider.generate`: generation via a fake chat client (JSON mode), failing closed on a client error.
- CLI: `propose` writes a staged file with provenance and does not touch `policies/`; `activate` promotes it, audits to the chain, and the chain still verifies `CHAIN_OK`.
- One `@pytest.mark.model`-gated real-model authoring test (skipped without `OPENAI_API_KEY`).
- Gates: coverage ≥ 85%, ruff, `mypy --strict`.

---

## 9. Scope boundary (explicit deferrals)

- No `injection_signatures` authoring (curated threat list, not NL-authored).
- No Warden discovery / intent-drift / embeddings (SP4); the tool inventory is supplied to `propose`, not auto-discovered yet.
- No dashboard authoring UI (SP6); the CLI is the human gate this cycle.

**Future work (not this cycle):** *adversarial policy testing* — after authoring, have the model generate example tool-calls the proposed policy would block/flag, so a reviewer sees the policy's real effect alongside the diff. Strengthens the human gate; deferred to keep this cycle focused.
